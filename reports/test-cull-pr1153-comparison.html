Executive Read

I agree with the broad direction of PR 1153: a lot of component-render, copy, fixture-shape, and string-mapping tests are low signal. The place I would be more conservative is around transaction flow, external URL validation, quote/slippage math, Safe status polling, rewards accounting, fee display, route normalization, and API response schema retention. Those tests protect user funds, user trust, or external integration boundaries in a way E2E smoke tests usually do not cover cheaply.

Recommendation: keep durable safety/math/integration-boundary unit tests, allow temporary PR tests during agent-assisted implementation, cull low-signal temporary tests before merge, and cover public user flows with E2E smoke tests.

Suggested Policy

  • Keep unit tests for money math, slippage, fees, rewards, URL safety, external API normalization, transaction state machines, Safe polling, and schema fields that protect app/backend contracts.
  • Cull tests that only assert static copy, component shell rendering, mocked hook wiring, temporary launch banners, or one-line boolean wrappers.
  • Convert user-facing flows to E2E smoke tests: vault discovery/search, vault detail rendering, connect-wallet affordances, deposit/withdraw quote display, and transaction overlay happy/error states.
  • Use bun run test:cull at the end of agent turns for changed tests, and use bun run test:cull -- --all sparingly as a periodic repo hygiene review.
